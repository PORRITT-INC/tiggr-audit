TIGGR — Tamper-Evident Audit of AI API Exchanges

A customer-controlled, hash-chained audit layer for AI workflows — validated on a live first-party desktop client
Timothy Porritt, Porritt Inc. (Layton, UT)  |  2026-06-20  |  Patent pending  |  SAM-registered (UEI WQWHVJ1J98S9)

TIGGR sits between an AI client and the model API and writes an append-only, cryptographically chained record of every request and response to a local database. The operator gets two synchronized views of the same traffic: a human-readable audit trail and a tamper-evident cryptographic record. Any post-hoc alteration of the log breaks the chain at the modified entry. The system has recorded 60,000+ exchanges in continuous field use and has been validated capturing the full payload of a live desktop-client conversation.

1. Two synchronized views of one captured exchange

Every intercepted request/response is written in two forms, from the same source event.

View A — Human-readable audit trail

Operators (and auditors) read traffic in a familiar flow format — endpoint, status, size:

2026-06-20T04:04:36Z   GET   https://api.anthropic.com/v1/models                 << 200 OK     614 B
2026-06-20T05:22:37Z   POST  https://api.anthropic.com/v1/messages?beta=true     >> request  3,688,162 B
2026-06-20T05:22:41Z   POST  https://api.anthropic.com/v1/messages?beta=true     << 200 OK   (streamed)
   request body (decoded):  {"model":"claude-opus-4-8","messages":[{"role":"user",
                             "content": [operator prompt — withheld for privacy] }, ... ]}

The decoded body contains the operator's verbatim prompt and the model's reply. Real content is withheld in this document; it is captured in full locally and is readable on demand for the data owner.

View B — Tamper-evident cryptographic record

The same events are also stored as an opaque, hash-chained ledger. Each row carries a SHA-256 digest of its payload and a chain value that binds it to the prior row:

id        timestamp_utc          dir       payload_sha256 (truncated)     chain (truncated)

Proprietary: the exact chain-construction algorithm (the TIGGR protocol) is intentionally not disclosed in this document. What is shown is the externally observable output — opaque per-record and chain digests — which is all an auditor needs to verify integrity without access to the method.

4. What makes it notable

  • Two views, one source of truth: readable for humans, cryptographically chained for verification — synchronized automatically.
  • Validated on a live first-party client, not a mock — full multi-megabyte payloads captured and verifiable.
  • Customer-controlled & local: no cloud, no third party; the data owner holds the only copy.
  • Independently verifiable: an auditor can re-run the verifier over the ledger and reproduce the integrity result without the proprietary method.
  • Pairs with Porritt's HPC→edge stack: models trained on a DOE supercomputer and run on sub-watt neuromorphic silicon, now with an audit-grade record of the AI exchanges around them.

5. Scope & privacy (stated plainly)

  • TIGGR audits the operator's own AI API usage — the operator's side of the wire. It does not claim visibility into the provider's internal systems.
  • Captured bodies contain real prompts and are kept local; this document deliberately shows redacted content and withholds the chain algorithm.
  • The hash chain attests that the record has not been altered after capture.
